# Code Review

## Overview

代码审查规范，涵盖审查流程、审查标准、评论规范和审查文化。适用于所有通过 PR / MR 进行的代码审查。

---

## Rules

### R01 — 审查流程

**MUST** — 代码审查必须遵循以下流程：

```
开发者提交 PR
    ↓
CI 自动检查（lint / test / build）
    ↓
至少 1 人 Review
    ↓
修改（如有）
    ↓
Approve
    ↓
合并
```

**CI 未通过的 PR，禁止人工 Review。** 避免浪费审查者时间。

---

### R02 — PR 粒度

**MUST** — PR 必须控制粒度：

| 指标 | 要求 |
|------|------|
| 代码行数 | ≤400 行（不含自动生成代码） |
| 文件数 | ≤10 个 |
| 功能点 | 1 个逻辑功能 |

**超过 400 行的 PR 必须拆分。** 大 PR 审查质量显著下降。

✅ Correct:

```
PR #101: feat(auth): add login endpoint
PR #102: feat(auth): add JWT token generation
PR #103: test(auth): add login tests
```

❌ Wrong:

```
PR #101: feat: add entire auth module (2000 lines, 30 files)
```

---

### R03 — PR 描述

**MUST** — PR 描述必须包含：

```markdown
## 变更内容
简要描述本次变更

## 变更原因
为什么需要这个变更

## 影响范围
影响哪些模块 / API / 数据库

## 测试方式
- [ ] 单元测试通过
- [ ] 手动测试通过
- [ ] 无破坏性变更

## 关联 Issue
Closes #123
```

---

### R04 — 审查者数量

**MUST** — 审查者要求：

| 场景 | 最低 Approve 数 |
|------|----------------|
| 普通功能 | 1 人 |
| 核心模块（支付/安全/数据） | 2 人 |
| 架构变更 | 架构师 + 1 人 |
| hotfix | 1 人（可事后补审） |

---

### R05 — 审查时效

**SHOULD** — 审查时效要求：

| 优先级 | 响应时间 |
|--------|---------|
| hotfix | 2 小时内 |
| 普通 PR | 1 个工作日内 |
| 大型 PR | 2 个工作日内 |

**超过时效，开发者有权提醒审查者。**

---

### R06 — 审查维度

**MUST** — 审查必须覆盖以下维度（按优先级排序）：

| 优先级 | 维度 | 关注点 |
|--------|------|--------|
| P0 | 正确性 | 逻辑是否正确、边界条件是否处理 |
| P0 | 安全性 | 是否有注入、越权、敏感信息泄露 |
| P1 | 可维护性 | 代码是否可读、命名是否清晰 |
| P1 | 性能 | 是否有 N+1、内存泄漏、慢查询 |
| P2 | 规范性 | 是否符合编码规范、API 规范 |
| P2 | 测试 | 是否有足够的测试覆盖 |
| P3 | 风格 | 代码风格、格式（可由 linter 自动处理） |

**P3 风格问题不应阻塞合并，应由 linter 自动处理。**

---

### R07 — 评论分类

**MUST** — 审查评论必须标注类型：

| 标签 | 含义 | 是否阻塞合并 |
|------|------|-------------|
| `[MUST]` | 必须修改 | 是 |
| `[SHOULD]` | 建议修改 | 否（需回复理由） |
| `[NIT]` | 小问题（命名、格式） | 否 |
| `[QUESTION]` | 疑问 | 否（需回复） |
| `[PRAISE]` | 好的实践 | 否 |

✅ Correct:

```
[MUST] 这里缺少空值检查，可能导致 NPE
[SHOULD] 建议使用 Optional 替代 null 返回
[NIT] 变量名建议用 userId 而非 uid
[QUESTION] 这个逻辑在并发场景下安全吗？
[PRAISE] 这个抽象做得很好，复用性很强
```

❌ Wrong:

```
这里有问题
改一下
不对
```

---

### R08 — 审查者行为规范

**MUST** — 审查者必须遵循：

- **对事不对人**：评论代码，不评论人
- **给出具体建议**：不只说"不好"，要说"怎么改"
- **解释原因**：不只说"改"，要说"为什么改"
- **区分必要和建议**：用标签区分 MUST / SHOULD / NIT
- **认可好的代码**：不只挑问题，也肯定好的实践

✅ Correct:

```
[MUST] 这里缺少空值检查。当 user 为 null 时会抛 NPE，
建议在 L42 添加 if (user == null) 的判断。
```

❌ Wrong:

```
这代码写得不行
你不知道要判空吗？
```

---

### R09 — 被审查者行为规范

**MUST** — 被审查者必须遵循：

- **不把 Review 当人身攻击**：审查是对代码的，不是对人的
- **回复每条评论**：修改了就标记 Resolved，不修改就说明理由
- **不争论 NIT**：小问题直接改，不值得争论
- **主动解释复杂逻辑**：在 PR 描述或代码注释中说明
- **及时修改**：收到评论后 1 个工作日内响应

---

### R10 — 自审

**MUST** — 提交 PR 前必须自审：

```markdown
## 自审清单
- [ ] 代码可编译、可运行
- [ ] 无调试代码（console.log / System.out / print）
- [ ] 无敏感信息（密码 / Token / 内网 IP）
- [ ] 无未使用的 import / 变量
- [ ] 测试通过
- [ ] 符合编码规范
- [ ] PR 粒度合适（≤400 行）
- [ ] PR 描述完整
```

---

### R11 — 合并条件

**MUST** — PR 合并必须满足所有条件：

- [ ] CI 通过（lint / test / build）
- [ ] 足够的 Approve
- [ ] 所有 `[MUST]` 评论已解决
- [ ] 所有 `[QUESTION]` 评论已回复
- [ ] 无未解决的合并冲突
- [ ] 分支已与目标分支同步

---

### R12 — 审查与重构分离

**SHOULD** — 审查和重构不应混在同一个 PR：

- 功能 PR：只做功能
- 重构 PR：只做重构
- 风格 PR：只做格式调整（由 linter 自动处理）

**禁止在功能 PR 中顺手重构无关代码。**

---

### R13 — AI 辅助审查

**MAY** — 可以使用 AI 辅助代码审查，但：

- AI 审查不能替代人工审查
- AI 审查结果必须人工确认
- AI 审查可作为 `[SHOULD]` 级别的建议
- 安全相关审查必须由人工完成

---

## Checklist

- [ ] CI 通过后才进行人工 Review
- [ ] PR 粒度 ≤400 行
- [ ] PR 描述完整（变更内容、原因、影响、测试）
- [ ] 审查者数量符合要求
- [ ] 审查覆盖正确性、安全性、可维护性
- [ ] 评论标注类型（MUST / SHOULD / NIT / QUESTION / PRAISE）
- [ ] 审查者对事不对人
- [ ] 被审查者回复每条评论
- [ ] 提交前已自审
- [ ] 合并条件全部满足
- [ ] 功能与重构分离
