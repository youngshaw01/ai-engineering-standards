# System-Design（系统设计）

## Overview

系统设计是从需求到实现的桥梁，涵盖架构选型、容量规划、可用性保障、扩展策略和容灾方案等核心环节。一份严谨的系统设计文档能够降低技术风险、指导团队协作，并为后续演进提供基线参照。本标准定义了系统设计过程中必须遵循的原则和实践方法。

---

## Rules

### R01 — 系统设计流程

**MUST** 在编写代码之前完成系统设计文档，明确目标、约束和非功能性需求。

**SHOULD** 采用"需求分析 → 接口定义 → 数据模型 → 架构设计 → 评审"的递进式流程。

**MAY** 使用 ADR（Architecture Decision Record）记录每个关键决策及其理由。

✅ 正确示例：
```
1. 需求分析：日均 100 万订单，峰值 QPS 5000
2. 接口定义：RESTful API + gRPC 内部调用
3. 数据模型：MySQL 分库分表 + Redis 缓存
4. 架构设计：微服务 + Kafka 异步解耦
5. 评审：团队 Review 并签字确认
```

❌ 错误示例：
```
直接开始写代码，边写边改，无统一设计文档
```

---

### R02 — 容量估算

**MUST** 对关键指标进行量化估算：QPS、存储量、带宽、并发连接数。

**SHOULD** 基于业务增长趋势预留 3-6 倍的容量余量。

**MAY** 建立容量模型，定期更新实际数据与估算值的差异。

✅ 正确示例：
```
日活用户：100 万
平均操作次数/人/天：20
峰值系数：3
峰值 QPS = 1,000,000 × 20 / 86400 × 3 ≈ 7,000 QPS
每条记录约 500B，年存储增量 = 100万×20×365×500B ≈ 3.6TB
```

❌ 错误示例：
```
"流量应该不大，先上线再说"
```

---

### R03 — 高可用设计（冗余与故障转移）

**MUST** 识别单点故障（SPOF），对所有关键组件实施冗余部署。

**SHOULD** 设计自动故障转移机制，MTTR（恢复时间）目标 < 5 分钟。

**MAY** 通过混沌工程验证故障场景下的系统行为。

✅ 正确示例：
```
应用层：至少 3 个实例，分布在不同可用区
数据库层：主从复制 + 自动切换（MHA / Orchestrator）
负载均衡层：跨 AZ 部署，健康检查自动摘除异常节点
```

❌ 错误示例：
```
所有服务实例部署在同一台物理机上
数据库只有单节点，无备份
```

---

### R04 — 可扩展性模式（水平/垂直/分片）

**MUST** 优先选择水平扩展方案，避免依赖单机性能上限。

**SHOULD** 当单表数据量超过阈值（如 5000 万行）时，实施分片策略。

**MAY** 使用一致性哈希或范围分片减少数据迁移成本。

✅ 正确示例：
```
-- 按 user_id 取模分片
CREATE TABLE orders_0 LIKE orders;
CREATE TABLE orders_1 LIKE orders;
...
-- 路由规则
shardIndex = userId % 16;
```

❌ 错误示例：
```
单库单表承载所有数据，查询越来越慢后只能停机迁移
```

---

### R05 — 一致性与分区容忍（CAP）

**MUST** 在分布式系统中明确选择 AP 或 CP 策略，并在设计文档中说明取舍理由。

**SHOULD** 对于非关键业务路径采用最终一致性以提升可用性；对于金融交易等强一致场景采用 CP。

**MAY** 使用版本向量（Version Vector）或 CRDT 解决冲突。

✅ 正确示例：
```
订单状态同步：AP（允许短暂不一致，最终一致）
支付扣款：CP（必须强一致，宁可短暂不可用）
```

❌ 错误示例：
```
所有场景都要求强一致，导致系统可用性极低
```

---

### R06 — 延迟优化

**MUST** 设定端到端延迟 SLA（如 P99 < 200ms），并逐层拆解耗时分布。

**SHOULD** 优先优化高频路径（Hot Path），将串行调用改为并行。

**MAY** 引入多级缓存（本地缓存 → 分布式缓存 → 数据库）降低响应时间。

✅ 正确示例：
```
P99 延迟分解：
  网络传输：20ms
  应用逻辑：50ms
  缓存读取：10ms
  数据库查询：80ms ← 瓶颈所在
优化：增加缓存命中率 + SQL 索引优化，降至 30ms
```

❌ 错误示例：
```
只关注整体延迟，不逐层排查，盲目加机器
```

---

### R07 — 灾难恢复（RPO/RTO）

**MUST** 明确定义 RPO（恢复点目标）和 RTO（恢复时间目标），并写入运维手册。

**SHOULD** 定期执行灾备演练，验证恢复流程的有效性。

**MAY** 跨地域部署实现异地多活，RTO 趋近于零。

| 场景 | RPO | RTO | 方案 |
|------|-----|-----|------|
| 普通业务 | < 5min | < 30min | 定时备份 + 主从切换 |
| 核心交易 | 0 | < 5min | 同步复制 + 异地双活 |

✅ 正确示例：
```
RPO: 0（同步复制到异地机房）
RTO: < 5min（自动故障检测 + 切换）
每月一次灾备演练，记录结果并改进
```

❌ 错误示例：
```
从未制定 RPO/RTO，灾备方案停留在纸面
```

---

### R08 — 设计文档模板

**MUST** 使用统一的设计文档模板，确保信息完整性和可追溯性。

**SHOULD** 包含以下章节：背景与目标、接口规范、数据模型、架构图、容量估算、风险评估、演进计划。

**MAY** 附上原型代码或 POC 验证结果。

✅ 正确示例：
```markdown
# 系统设计文档
## 1. 背景与目标
## 2. 接口规范
## 3. 数据模型
## 4. 架构设计
## 5. 容量估算
## 6. 风险评估
## 7. 演进计划
## 附录：POC 验证结果
```

❌ 错误示例：
```
口头传达设计方案，无书面记录
```
